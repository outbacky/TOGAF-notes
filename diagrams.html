01

What is TOGAF?

Framework overview & four domains

TOGAF (The Open Group Architecture Framework) is a globally recognised framework for practising Enterprise Architecture — a structured, repeatable approach to designing and governing an organisation's business and technology landscape.

At its core, TOGAF organises an enterprise into four architecture domains (BDAT) and provides a methodology, vocabulary, and set of best practices that can be adapted to any organisation.

02

The Architecture Development Method

ADM Phase Wheel — the core iterative cycle

The ADM is a structured, iterative cycle of phases. Rather than a one-time project, it is designed to repeat continuously as the organisation evolves. Requirements Management sits at the centre — not a phase itself, but a continuous process connecting every phase.

ADM Iteration Types

Not every change requires running all phases from scratch. Architects re-enter the cycle at whichever iteration level fits the scale of the change.

Architecture Abstraction Levels

Contextual → Conceptual → Logical → Physical

Different stakeholders need different levels of detail. TOGAF defines four abstraction levels — a progression from the most abstract (big picture) to the most concrete (implementation-ready). Each level builds on the one above it.

LevelKey QuestionPrimary ADM Use
ContextualWhy?Phase P & A — scope & motivation
ConceptualWhat?Phase A & B — service models
LogicalHow organised?Phases B, C, D — architecture design
PhysicalWith what?Phases E & F — solution realisation

Architecture Building Blocks

ABBs & SBBs — reusable, composable units of architecture capability

Architecture can be assembled from reusable components rather than designed from scratch every time. An ABB defines what is needed (logical); an SBB defines how it is built (physical). The same ABB can be fulfilled by different SBBs in different contexts.

07

RAW vs. SAW

Scoping & filtering architecture work

RAW (Required Architecture Work) is the full, unfiltered picture of everything that could be done. SAW (Scoped Architecture Work) is the realistic, approved subset — filtered through stakeholder negotiation, principles, and trade-off analysis.

SLA BB — the Service Level Agreement Building Block is formally defined during RAW → SAW scoping, establishing performance and service expectations for each architectural component.
08

Change Requests Driving ADM Cycles

Eight sources that trigger new ADM cycles

New architecture work is continuously initiated by change requests from a wide variety of sources — some top-down (board strategy), some bottom-up (developer discovery), some external (regulation, market shift). All eight ultimately feed into the same ADM process.

09

Phase E vs. Phase F

Opportunities & Solutions vs. Migration Planning

Phase E is analytical — surveying possible solutions, checking interoperability, selecting the best building blocks. Phase F is a planning phase — taking Phase E decisions and sequencing them into a concrete roadmap. Skipping E risks committing to a plan built on incompatible components.

SBB Compatibility Stack

Phase E evaluates solutions preferring generic, reusable components before committing to custom builds.

Agile EA vs. Traditional EA

Choosing the right delivery approach for the project

TOGAF is not inherently waterfall. It supports both predictive and agile delivery styles. The decision comes down to two axes: how frequently the organisation needs to deliver changes, and how much the architecture changes with each delivery.

ApproachBest ForADM Style
PredictiveLarge infrastructure, low iterationFull ADM cycle, single pass
Agile / IterativeSoftware-intensive, high-changeIncremental ADM iterations
Transition-basedBridge between bothMultiple defined increments T1, T2, T3
12

Enterprise Security Architecture

Security as a parallel, continuous track across all ADM phases

Security is not a separate phase — it is a parallel, continuous track that runs alongside every ADM phase. At each phase the security architect asks: what risks are we introducing, and what controls are needed? By the time architecture reaches implementation, security has been designed in — not patched on.

13

Repository & Enterprise Continuum

Institutional memory & the building block classification system

The Architecture Repository is the EA function's institutional memory — storing all artefacts across every ADM cycle. The Enterprise Continuum classifies building blocks on a spectrum from generic (Foundation) to organisation-specific, enabling systematic reuse.

14

Business Scenarios

Translating stakeholder concerns into architecture requirements

Business Scenarios prevent the most common architecture failure — designing a solution to the wrong problem. They are a structured conversation between the Architecture Team and business stakeholders that produces requirements across all five domains before any design decisions are made.

18

Classes of Architecture Engagement

Calibrating the level of EA involvement to the nature of the change

Not all architecture work warrants a full enterprise programme. The engagement class is determined by three questions — why the change is occurring, when and how often it is needed, and what the scope is. The answers determine which ADM phases to emphasise and how formal the governance should be.

The engagement class determines how the ADM is applied — from lightweight advisory work on a small initiative through to a full enterprise re-architecture programme spanning multiple years.
